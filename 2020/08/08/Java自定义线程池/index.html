四种拒绝策略

其实注释里写的很详细了，简单翻译一下

AbortPolicy(default handler)

这是默认的拒绝策略，若所有线程都已经占用，且工作队列（有界）满了，则新进来的任务会被直接直接丢弃并抛出异常（不影响已有的任务）

联系到代码就是，当maximumPoolSize+Queue的界（长度）全被任务占满时，再新来任务都会被抛弃并抛出异常

我给maximumPoolSize+Queue的界（长度）自己定一个名字叫“工作空间”好了，方便表达

CallerRunsPolicy

当工作空间被占满时，新来的任务不会被交给线程池执行，而是会被交给调用线程池的线程执行，例如，主线程调用线程池执行System.out.println("Hello World")，但线程池的工作空间满了，那么线程池就告诉主线程：自己执行去吧！于是主线程会自己执行打印这个Hello World

DiscardPolicy

当工作空间被占满时，新来的任务会被直接丢弃，不抛出异常，不影响正在执行的任务和等待的任务

DiscardOldestPolicy

当工作空间被占满时，新来的任务不会被直接丢弃，会丢弃未被执行的最早进来的任务，也就是WorkQueue会执行poll()操作，弹出最先进来的任务，然后执行新进来的任务，这个可以在代码中体现出来

执行结果及分析

这是其中一种执行结果，根据CPU心情执行结果可能不同，下面分析下执行的流程

  • 主线程通过调用线程池开启了5个任务

  • 任务1开启，线程池为其分配一个线程，此时挂起1秒。执行打印语句输出队列长度为0，线程池大小为1

  • 任务2开启，发现无可用线程，进入队列等待。执行打印语句输出队列长度为1，线程池大小为1

  • 任务3开启，发现无可用线程，进入队列等待。执行打印语句输出队列长度为2，线程池大小为1
  • 任务4开启，发现无可用线程，进入队列等待。执行打印语句输出队列长度为3，线程池大小为1
  • 任务5开启，发现无可用线程，队列已满，线程池开启备用线程执行任务5，执行打印语句输出队列长度为3，线程池大小为2
  • 任务1执行完毕，打印1
  • 任务5执行完毕，打印5，这是因为2，3，4都在队列里等待执行，而5来了没排队就被分配线程执行了，所以在前
  • 任务2执行完毕，打印2
  • 任务3执行完毕，打印3
  • 任务4执行完毕，打印4

可以发现规律是，当队列满了以后，如果有未被开启的线程，新来的任务会插队，由备用线程执行。
